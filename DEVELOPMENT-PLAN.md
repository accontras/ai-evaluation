## 为什么换路线？

v2 计划是"42 个 session → 3 个月 → v1.0.0"。实际跑完 M0-M3 后发现几个问题：

1. **业余 session ≠ 工作 session**。晚上 2-3 小时的有效编码时间约 1-1.5 小时（需要切上下文、回顾上次进度）。计划里写 3h 的 session，实际要 4-6h。
2. **残留的 12 个 session 塞了太多东西**。多模型对比、NL→JEXL、RAG 检索——每个都是独立的学习课题，3 小时完全不够做深。
3. **"赶 checkmark"和"提能力"是两件事**。42 个 session 赶完，你会得到一个可以展示的项目，但每个 AI 工程化能力都是浅的——跑通了，不理解。
4. **你的真实需求不是"发布一个开源项目"**。你是 Java 架构师往 AI 方向转，需要的是**可迁移的 AI 工程能力**——不是多一个 GitHub repo，是多一项简历上能写深度的技能。

v3 的核心变化：

```
v2:  S31→S32→...→S42 → v1.0.0 (赶 checkmark)
v3:  A1(Prompt工程) → A2(可观测性) → A3(RAG) → A4(可靠性) → A5(发布)

     每个模块做完，你在这件事上就是"做过的人"，不是"看过教程的人"。
```

## A1: Prompt 工程体系（3-4 周）🔄

> 核心认知: Prompt 不是"写一段话"。Prompt 是代码——可测试、可版本化、可对比、可回滚。

### A1.1 多模型对比框架
**时间**: 2-3 个 session（~8h）

```
□ 整合 2-3 个模型 provider (DeepSeek + 至少 1 个备选)
  - 统一 LlmClient 接口: chat(prompt, model) → LlmResponse
  - 每个 provider 的差异封装在适配层
□ 同一评估任务 → N 个模型各自打分
□ 对比维度:
  - 打分一致性: 同一指标不同模型差多少？
  - 评分稳定性: 同一 prompt 跑 3 次，方差多大？
  - 与规则引擎基准的偏差
□ API: POST /api/v1/ai-experiments/model-compare
  - 返回: { indexCode, modelScores: { deepseek: 85, glm: 72, qwen: 80 }, ruleScore: 70, variance: ... }
```

**验证**: 3 个模型 × 3 个指标 × 3 次重复 → 有统计学意义的对比数据

**AI 工程化收获**: 
- 多模型差异的直观感受
- 知道"一致性"和"稳定性"是两个不同维度
- 能说出 DeepSeek/GLM/Qwen 在不同指标类型上的表现差异

### A1.2 Prompt 版本化管理
**时间**: 2 个 session（~6h）

```
□ PromptTemplate 版本化:
  - 每个 prompt 有 version + changelog
  - 数据库/文件存储，可追溯
  - 评估记录关联 promptVersion → 事后可复盘
□ Prompt 效果评估:
  - 对同一个指标，用 3 个不同版本的 prompt 打分
  - 对比: 哪个版本更接近规则引擎基准？哪个更稳定？
  - eval_ai_experiment.prompt_version 字段
□ API: GET /api/v1/ai-experiments/prompt-history?indexCode=xxx
  - 返回: [{ version, avgScore, avgDiffFromRule, sampleCount, createdAt }]
```

**验证**: 同一个指标 → 2 个 prompt 版本 → 效果数据可对比

**AI 工程化收获**:
- Prompt 迭代不再靠记忆和直觉
- 能回答"v2 比 v1 改进了什么"而不是"我感觉改好了"
- 这套方法论可以迁移到任何 LLM 应用

### A1.3 自动评测 + 数据积累
**时间**: 2 个 session（~6h）

```
□ 跑 50 次评估（手动编不同场景的数据）

  - 哪些指标 LLM 和规则引擎分歧最大？
  - 不同 prompt 版本对分歧的影响？
  - 不同模型在"需要上下文理解"vs"纯数值判断"的指标上差异？
□ 输出: 一份 Prompt 工程实验报告（不是你给别人的，是你自己的认知沉淀）
  - 写到 wiki/research/ 或 eval-system/docs/
  - 格式不限，但要能回答"你从这组实验里学到了什么"
```

**验证**: 50+ 条实验数据，有明确的实验结论

## A3: RAG 检索（3-4 周）✅ (2026-07-21/22)

> 核心认知: RAG 不是"接个向量数据库"。RAG 是一整条 pipeline——embedding 怎么选、chunk 怎么切、检索怎么评、如何注入 prompt——每一步都有工程决策。

### A3.1 Embedding + 向量存储 ✅ (2026-07-21)
**实际耗时**: 1 个 session（~4h 编码 + 踩坑）

```
✅ 选型评估:
  - Embedding 模型: bge-small-zh-v1.5 (BAAI, 512维, ONNX 90MB, 本地 CPU 推理)
  - 向量存储: Lucene HNSW (纯 Java, 零外部依赖)
  - 决策依据: 本地部署/数据不出域/零 API 费用/Windows 兼容
  - 详细选型: wiki/ai/RAG-基础知识-从零到懂.md
✅ 历史评估记录 → Embedding → 向量存储:
  - 对指标名+实际值+LLM打分理由做 embedding
  - 启动时异步迁移 126 条历史记录（37秒）
  - 每次评估完成后增量写入索引
✅ 实现:
  - EmbeddingService: ONNX Runtime 直连推理, mean pooling + L2 归一化
  - VectorIndexService: Lucene 9.x KnnVectorField HNSW 索引
  - VectorRagService: encode(3-5ms) + search(6-11ms), 总延迟 ~10-15ms
  - RagIndexInitializer: 异步全量迁移 + 增量钩子
  - LlmScoringStrategy: 向量优先 → 规则降级 → 空 safe fallback 三层防护
✅ 落地指南: wiki/ai/RAG-落地实践指南.md
```

**验证**: ✅ 126 条历史记录 → embedding → 向量检索可用, 端到端评估通过

**坑点记录**:
- Maven 阿里云镜像劫持: `mirrorOf` 从 `*` 改为 `*,!central`
- DJL `djl-core` artifact 不存在 → 改用 `ai.djl:api`
- DJL `ModelZoo.loadModel()` 不支持本地 ONNX → 换 ONNX Runtime Java API 直连
- ONNX 模型需 `token_type_ids` 输入（代码只传了 input_ids + attention_mask）
- `Lucene99Codec` 类在 Lucene 9.12 中已移除

### A3.2 检索 + 注入 Prompt ✅ (2026-07-21)

```
✅ 检索策略:
  - 对每个指标分别向量检索 Top-2, 合并去重取 Top-3
  - 向量无结果 → 降级到 SimilarCaseService 规则检索
✅ 注入 LLM Prompt 作为 few-shot:
  - "## 语义相似历史案例（向量检索）" 段落
  - 案例含: 指标名 + 得分 + 理由 + 相似度%
✅ 已集成进 LlmScoringStrategy.buildFewShot() → H3 Pipeline
```

**验证**: ✅ 评估对象 → 自动向量检索 3 个相似案例 → 注入 prompt → LLM 打分参考

### A3.3 检索质量评测 ✅ (2026-07-22)

**实际耗时**: 1 个 session（编码） + 跑批

```
✅ 标注数据集:
  - 来源: 168 条历史评估日志 (eval_indicator_log)
  - 规模: 15 条标注查询 (10 条代表性 + 5 条边界)
  - 覆盖: 正常值 / 边界值 (零值/极端高值/无数据/负值/完美值)
  - 标注: 二元相关 (1/0), 人工判断
✅ 对比评测 (向量 vs 规则):
  - 向量: bge-small-zh-v1.5 embedding → Lucene HNSW KNN
  - 规则: 指标编码匹配 + 得分接近度 + diff 级别加权
✅ 评测指标: Hit Rate@K + NDCG@K (K=1,3,5)
✅ 可视化: eval-system/data/rag-eval-results/index.html
```

#### 跑批结果 (2026-07-22)

| 指标 | 通道 | K=1 | K=3 | K=5 |
|------|------|-----|-----|-----|
| **Hit Rate** | 向量 | 20.0% | 33.3% | **53.3%** |
| | 规则 | 26.7% | 26.7% | 26.7% |
| **NDCG** | 向量 | 100% | 95.0% | 82.8% |
| | 规则 | 100% | 98.9% | 98.7% |

#### 关键发现

1. **向量检索召回率随 K 增长翻倍**（HR@1=20% → HR@5=53.3%）：语义检索能找到规则匹配不到的相关案例，但排序精度有限
2. **规则检索 HR 恒定 26.7%**：只匹配同 index_code 的历史记录，无法跨指标检索
3. **NDCG 揭示排序质量**：向量 Top-1 完美（100%），但 Top-5 降到 82.8%——相关案例没有完全排在前面
4. **向量更适合高召回、跨指标场景**；规则更适合精确匹配、排序稳定场景
5. **15 条标注偏少**，统计显著性不足，但趋势明确

**实验笔记**: `wiki/research/RAG-检索质量评测-20260722.md`

## 附录：可选探索项（不排工期）

### NL → JEXL：用自然语言写规则表达式

**是什么**：业务人员用自然语言描述一条规则，LLM 自动翻译成 JEXL 表达式。

```
输入（自然语言）: "物流费用偏差超过20%就是红线"
输出（JEXL）:      attrValues["costDeviation"] > 20

输入: "入职天数小于30天的新员工，合格线降到60分"
输出: attrValues["daysSinceHire"] < 30 ? 60 : standardScore
```

**为什么放附录**：它是一个 **code generation** 任务，不是 AI 工程化基础能力。先理解 Prompt 工程、可观测性、RAG 之后，回头看这个会有更准确的判断——哪些能做、哪些难做、为什么难做。

**如果想探索**（不设时间表）：
- 先手写 10 条自然语言规则 → 让 LLM 试试转换 → 看准确率
- 理解 JEXL 沙箱安全：生成的表达式不能随意执行（`RuntimeException` 要在沙箱里跑）
- 核心难点不在"翻译"，在"变量名映射"——LLM 怎么知道 `costDeviation` 对应哪个字段？

**参考方向**：Text-to-SQL、GitHub Copilot 的代码补全——本质都是 NL → 结构化语言，工程上已经成熟，但领域特定（NL→JEXL）需要自己探索。

```
│              └────────────────┘                   │
└──────────────────────────────────────────────────┘

分工铁律:
  LLM 做: 语义判断 (打分、异常检测、总结)
  规则引擎做: 确定性运算 (聚合、排名、等级区间)
  永不混淆: 不让 LLM 做数学，不让规则引擎做判断
```

## AI 替代了什么？（一张表说清楚）

| Handler | AI 替代? | 为什么 |
|---------|---------|--------|
| H1 加载配置 | ❌ | 读数据库，AI 没价值 |
| H2 提取指标值 | ❌ | 字段映射，确定性操作 |
| **H3 标准匹配+打分** | **★ LLM-as-Judge (默认)** | 语义理解，上下文判断 |
| H3 派生指标 | ❌ | 数学运算，LLM 会算错 |
| **H3 树聚合** | **❌ 永不替代** | 加权求和必须确定，这是审计底线 |
| **H4 事件/红线** | **★ LLM 语义检测 (默认)** | 异常不靠穷举规则 |
| H5 申诉改分 | ❌ | 人工决策，系统只执行 |
| H6 等级映射 | ❌ | 分数→等级是查表 |
| **H6 总结** | **★ LLM 多轮对话** | 生成+自审 |
| Ranking | ❌ | 数学排序 |

**三个 AI 切入点，恰好是评估系统里最有判断力的三个环节。**

## 关键提醒

1. **不做 NL→JEXL**。它是个 research problem，不是 AI 工程化能力训练场。投入产出比太低。

2. **每个模块必须有实验笔记**。不是代码跑通了就算完——你从实验数据里学到了什么？写到 wiki/research/ 下。这是你半年后复习时最值钱的东西。

3. **对比数据是你最值钱的资产**。A1 的多模型对比 + A3 的检索评测——这两组数据在任何 AI 项目上都是硬通货。

4. **树聚合永不交给 LLM**。审计底线，不变。

5. **发布是能力的副产品**。不是"为了发布而赶工"，而是"能力到了，发布是水到渠成"。v1.0.0 什么时候发，取决于 A1-A4 什么时候做完，不是日历上的日期。

6. **允许中断，不允许敷衍**。出差一周、加班三天、孩子生病——都正常。回来继续就行。但每个 session 的代码不能为了"赶上进度"而写烂。
