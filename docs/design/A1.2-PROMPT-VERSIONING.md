# A1.2: Prompt 版本化管理 — 设计文档

> **状态**: 待审批 | **日期**: 2026-07-21
> **核心认知**: Prompt 是代码——可测试、可版本化、可对比、可回滚。

---

## 一、为什么需要版本化？

当前状态：
- `LlmScoringStrategy` 写死硬编码的 `SYSTEM_PROMPT` 和 `USER_PROMPT_TMPL`
- 每次改 Prompt 需要改 Java 代码 + 重新编译部署
- 无法回答"v3 比 v2 好在哪里"

目标：
- Prompt 存储在数据库/文件，运行时加载
- 每次 LLM 调用记录 `promptVersion`（已在 A2 的 `eval_ai_experiment` 表就绪）
- 同指标不同版本 Prompt 的效果可对比
- 版本间可切换、可回滚

---

## 二、存储方案

### 方案选择：数据库表 > YAML 文件 > 硬编码

| 方案 | 优点 | 缺点 | 选择 |
|------|------|------|------|
| 数据库表 | 可运行时增删改，自动关联实验记录 | 需要 DDL + CRUD | ✅ 选定 |
| YAML 文件 | 简单，版本控制友好 | 修改需重启，无运行时切换 | ❌ |
| 硬编码 | 零开销 | 改一次编译一次，无法对比 | ❌ 当前 |

### DDL

```sql
CREATE TABLE eval_prompt_template (
    id              BIGINT PRIMARY KEY AUTO_INCREMENT,
    prompt_key      VARCHAR(64)  NOT NULL COMMENT 'SCORING_SYSTEM / SCORING_USER / EVENT_DETECT / SUMMARY_R1',
    version         VARCHAR(32)  NOT NULL COMMENT 'v1-base / v2-standards / v3-fewshot',
    template_text   TEXT         NOT NULL COMMENT 'Prompt 模板文本',
    description     VARCHAR(500) COMMENT '版本说明/changelog',
    is_active       TINYINT DEFAULT 1 COMMENT '是否启用',
    created_at      DATETIME DEFAULT CURRENT_TIMESTAMP,
    UNIQUE KEY uk_key_version (prompt_key, version)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='Prompt 版本管理';
```

### 种子数据

当前活跃版本直接迁移现有硬编码 Prompt：

| prompt_key | version | template_text 内容 |
|------------|---------|-------------------|
| SCORING_SYSTEM | v3-fewshot | 当前 LlmScoringStrategy.SYSTEM_PROMPT |
| SCORING_USER | v3-fewshot | 当前 LlmScoringStrategy.USER_PROMPT_TMPL |
| SCORING_SYSTEM | v2-standards | 含标准注入但无 few-shot 的版本 |
| SCORING_USER | v2-standards | 含标准注入但无 few-shot 的版本 |

## 三、核心流程

### 3.1 运行时加载

```
LlmScoringStrategy.scoreAll()
  │
  ├─ promptVersion = resolveVersion("SCORING_SYSTEM", ctx)
  │     └─ 查 eval_prompt_template WHERE prompt_key=? AND is_active=1 ORDER BY version DESC LIMIT 1
  │
  ├─ systemPrompt = loadPrompt("SCORING_SYSTEM", promptVersion)
  ├─ userPrompt = loadPrompt("SCORING_USER", promptVersion).render(vars)
  │
  ├─ resp = llmClient.chatForJson(systemPrompt, userPrompt)
  │
  └─ recordExperiment(ctx, resp, count, promptVersion)
       └─ eval_ai_experiment.prompt_version = "v3-fewshot"
```

### 3.2 版本对比流程

```
同一条指标数据
  ├─ 用 v2-standards 打分 → score_v2
  ├─ 用 v3-fewshot   打分 → score_v3
  ├─ 用 v1-base      打分 → score_v1
  │
  └─ 对比:
       - 哪个版本更接近规则引擎基准？
       - 哪个版本的标准差更小（更稳定）？
       - 哪个版本的理由更具体？
```

## 四、API 设计

### 4.1 Prompt 版本 CRUD

| 端点 | 方法 | 说明 |
|------|------|------|
| `/api/v1/prompts` | GET | 列出所有 Prompt 模板 |
| `/api/v1/prompts` | POST | 新增一个版本 |
| `/api/v1/prompts/{id}` | PUT | 更新模板文本 |
| `/api/v1/prompts/{id}/activate` | POST | 激活该版本（同时禁用同 key 其他版本） |

### 4.2 Prompt 效果分析

| 端点 | 方法 | 说明 |
|------|------|------|
| `/api/v1/prompts/compare` | POST | 同指标用多版本打分对比 |

请求体：
```json
{
  "promptKey": "SCORING_USER",
  "versions": ["v1-base", "v2-standards", "v3-fewshot"],
  "testData": { "bizId": "TEST", "data": { "cost_deviation": 8.0, "abnormal_count": 2, "fill_rate": 88.0 } }
}
```

响应：
```json
{
  "versions": {
    "v1-base":      { "COST_DEV": 65, "ABNORM_CNT": 90, "FILL_RATE": 70 },
    "v2-standards": { "COST_DEV": 75, "ABNORM_CNT": 80, "FILL_RATE": 75 },
```

## 六、关键设计决策

### 6.1 为什么用数据库而不是文件？

- **运行时切换**: 不需要重新编译部署
- **自动关联实验记录**: `eval_ai_experiment.prompt_version` 已经就绪
- **与规则配置统一**: 评分标准 (`eval_model_standard`) 也在数据库

### 6.2 为什么每个 prompt_key + version 独立一行？

- 版本间独立，不会误改其他版本
- `is_active` 字段支持运行时切换
- UNIQUE KEY 防止版本号重复

### 6.3 为什么不搞完整的 "Prompt 工作流"（草稿→审核→发布）？

- 这是**个人开发项目**，不需要审批流
- 简单的 activate/deactivate + CRUD 足够
- 真正需要的是**效果数据对比**，不是流程管控
