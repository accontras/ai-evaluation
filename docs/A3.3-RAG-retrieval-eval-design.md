# A3.3 RAG 检索质量评测 — 设计文档

> 状态: ✅ 已完成 | 日期: 2026-07-22 | 版本: v1.0

---

## 1. 概述

### 1.1 要解决的问题

A3.1 和 A3.2 让 RAG pipeline **跑起来了**——embedding → Qdrant 检索 → 注入 Prompt。但一个关键问题没回答：

> **向量语义检索比规则特征检索到底好多少？在什么场景好，什么场景差？**

没有量化数据，"用向量还是用规则"就是一个信仰问题，不是工程决策。

A3.3 的目标：**用标注数据集 + 标准信息检索指标，量化回答这个问题。**

### 1.2 核心设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 标注方式 | 人工二元相关（1/0） | 标注成本可控，15 条足够验证方向 |
| 评测指标 | HR@K + NDCG@K | IR 领域标准指标，可对比业界基线 |
| 对比维度 | 向量 vs 规则 | 回答"要不要切"的核心问题 |
| 执行方式 | CLI runner（`rag-eval` profile） | 独立于主应用，不影响线上评估 |
| 结果持久化 | DB（eval_rag_compare_log）+ JSON 文件 | DB 可追溯，JSON 便于可视化 |

### 1.3 系统位置

```
eval-system/
├── eval-boot/src/main/java/.../runner/
│   └── RagQualityEvalRunner.java         ← 跑批入口（@Profile("rag-eval")）
├── eval-application/src/main/java/.../service/
│   ├── RagQualityService.java            ← 指标计算引擎（纯函数，无状态）
│   └── RagCompareTracker.java            ← 影子模式追踪器（内存 + DB）
├── eval-application/src/main/java/.../strategy/
│   └── LlmScoringStrategy.java           ← 生产代码中的影子模式钩子
├── eval-application/src/test/java/.../service/
│   └── RagQualityServiceTest.java        ← 指标计算单元测试（8 个 case）
└── data/

    └── rag-eval-results/
        ├── results.json                  ← 跑批结果
        └── index.html                    ← 可视化报告
```

## 2. 标注数据集

### 2.1 设计原则

1. **代表性优先于规模**：15 条精心挑选 > 100 条随机抽
2. **覆盖正常 + 边界**：正常值验证基本能力，边界值暴露系统盲区
3. **二元相关**：1=相关，0=不相关。降低标注主观性——"这个案例可以作为参考吗？"
4. **人工标注**：由开发者判断历史案例对当前查询的参考价值，不做自动标注

### 2.2 数据来源

从 `eval_indicator_log` 表中 168 条历史评估日志中筛选。这些日志包含三个指标：

| 指标编码 | 指标名称 | 记录数 | 值域 | 典型场景 |
|---------|---------|--------|------|---------|
| COST_DEV | 成本偏差 | ~60 条 | -5% ~ 25% | 成本控制评估 |
| FILL_RATE | 填充率 | ~60 条 | 80% ~ 100% | 物流效率评估 |
| ABNORM_CNT | 异常数量 | ~48 条 | 0 ~ 5 | 质量异常评估 |

### 2.3 数据集结构

```json
{
  "id": "gt-001",
  "indexCode": "COST_DEV",
  "indexName": "成本偏差",
  "dataValue": "17",
  "relevantLogIds": [40, 55, 112, 127],
  "notes": "17%偏差，预期命中其他15-20%的COST_DEV记录(NOTABLE)"
}
```

- `id`: 唯一标识，`gt-001` ~ `gt-015`
- `indexCode` / `indexName`: 查询的指标
- `dataValue`: 实际值（字符串，支持"无数据"等非数值）
- `relevantLogIds`: 人工标注的相关案例 log ID 列表
- `notes`: 标注理由和预期行为

### 2.4 查询覆盖矩阵

#### 代表性查询（10 条）——验证基本检索能力

| ID | 指标 | 值 | 标注特点 | 相关案例数 |
|----|------|-----|---------|-----------|
| gt-001 | COST_DEV | 17 | 正常偏差值，NOTABLE 级别 | 4 |
| gt-002 | FILL_RATE | 88 | 偏低填充率，NOTABLE | 4 |
| gt-003 | ABNORM_CNT | 1 | 低异常，SIGNIFICANT | 4 |
| gt-004 | FILL_RATE | 95 | 高填充率，TRIVIAL | 4 |
| gt-005 | ABNORM_CNT | 0 | 完美值(零异常)，LLM给100但规则给0 | 4 |
| gt-006 | COST_DEV | 20 | 正常偏差值(同 gt-001 相似) | 4 |
| gt-007 | FILL_RATE | 82 | 低填充率 | 4 |
| gt-008 | ABNORM_CNT | 2 | 中等异常 | 4 |
| gt-014 | FILL_RATE | 100 | 完美填充率 | 4 |
| gt-015 | ABNORM_CNT | 5 | 高异常值 | 3 |

#### 边界查询（5 条）——暴露系统盲区

| ID | 指标 | 值 | 标注特点 | 相关案例数 |
|----|------|-----|---------|-----------|
| gt-009 | COST_DEV | 0 | 零偏差。与 NULL 不同，历史无对应 | **0** |
| gt-010 | FILL_RATE | 999 | 极端高值，无真实数据对应 | **0** |
| gt-011 | ABNORM_CNT | "无数据" | 非数值。仅靠指标名匹配 | **0** |
| gt-012 | NEW_METRIC_RARE | 50 | **历史不存在的指标** | **0** |
| gt-013 | COST_DEV | -5 | 负值偏差，无对应 | **0** |

> **设计意图**: 边界查询的 `relevantLogIds` 设为空数组 `[]`——不是"不知道怎么标"，而是"确实没有高度相关的案例"。这检验系统能否在应该"找不到"的时候正确地找不到（而不是强行返回不相关的）。

## 3. 评测指标

### 3.1 Hit Rate@K（命中率）

**定义**：Top-K 检索结果中**至少包含一个**相关文档的查询占比。

```
HR@K = (包含 ≥1 个相关文档的查询数) / 总查询数
```

**计算逻辑**（见 `RagQualityService.hrAtK()`）：

```
for each query:
    for i in 0..min(K, resultCount):
        if relevance[i] == 1:
            hitCount++
            break  ← 只要有一个命中就计数
HR@K = hitCount / totalQueries
```

**物理意义**：衡量"用户翻前 K 个结果，能不能找到想要的东西"。HR@1 是最高标准——Top-1 就命中。

### 3.2 NDCG@K（归一化折损累计增益）

**定义**：考虑排序位置权重的检索质量指标。排名越靠前权重越高。

**DCG（折损累计增益）**：
```
DCG@K = Σ(i=0..K-1) rel_i / log₂(i + 2)
```
其中 `i+2` 对应 rank = i+1 → 折损因子 log₂(rank+1)：

| Rank | 折损因子 log₂(rank+1) |
|------|----------------------|
| 1 | log₂(2) = 1.00 |
| 2 | log₂(3) = 1.58 |
| 3 | log₂(4) = 2.00 |
| 4 | log₂(5) = 2.32 |
| 5 | log₂(6) = 2.58 |

**IDCG（理想 DCG）**：假设所有相关文档排在最优位置（最前面）。

```
IDCG@K = Σ(j=0..min(relevantCount,K)-1) 1.0 / log₂(j + 2)
```

**NDCG**：
```
NDCG@K = DCG@K / IDCG@K
```

**边界处理**：DCG=0 且 IDCG=0 时（所有结果都不相关），NDCG=1.0——"没有该返回的都没返回"是正确的。

**物理意义**：衡量排序质量。HR 只看"有没有"，NDCG 看"排得对不对"。

### 3.3 为什么选这两个指标

| 指标 | 回答的问题 | K 取值 |
|------|-----------|--------|
| HR@K | "能找到吗？" | K=1 最严格，K=3 常用，K=5 宽松 |
| NDCG@K | "排对了吗？" | 同上 |

两者互补：HR 高但 NDCG 低 → 能找到但排在后面。HR 低但 NDCG 高 → 命中率不行但命中的那条排序好。

### 3.4 二元相关的取舍

使用二元相关（1/0）而非多级相关（如 0-3）的原因：

1. **标注成本低**：判断"有没有参考价值"比"有多大参考价值"快得多
2. **主观性低**：多级标注的标准难以一致
3. **15 条规模下足够**：验证方向不需要精细的相关度分级
4. **可升级**：未来如果标注数据积累到 50+，可以引入分级相关度

### 4.2 启动方式

```bash
# 独立 profile，不影响主应用
cd eval-system
mvn spring-boot:run -Dspring-boot.run.profiles=rag-eval

# 或 jar 包
java -jar eval-boot.jar --spring.profiles.active=rag-eval
```

启动后自动执行全流程 → 输出结果 → `System.exit(0)`。

### 4.3 三层防护

```
Layer 1: Qdrant 可用性检查
  → Qdrant 不可用 → System.exit(1)，不执行评测

Layer 2: Ground truth 文件检查
  → 文件不存在或格式错误 → System.exit(1)

Layer 3: 每条查询独立异常隔离
  → 单条检索异常不影响其他查询（try-catch 包裹）
```

## 5. 影子模式集成

### 5.1 为什么需要影子模式

A3.3 不仅是"一次性跑批"——它同时在生产代码中织入了持续对比能力。每次 LLM 评估发生时，`LlmScoringStrategy.buildFewShot()` 自动双跑向量检索和规则检索，并通过 `RagCompareTracker` 记录对比统计。

```
LlmScoringStrategy.buildFewShot()
  ├── if vectorRagService.isAvailable():
  │     ├── safeBuildFewShotVector(ctx)     ← 向量检索
  │     ├── safeBuildFewShotRule(ctx)       ← 规则检索（并行降级）
  │     ├── compareAndLog(ctx, v, r)        ← 影子模式：记录对比
  │     └── return 向量结果 ?? 规则结果     ← 向量优先
  └── else:
        └── safeBuildFewShotRule(ctx)        ← 纯降级
```

### 5.2 RagCompareTracker 设计

```
RagCompareTracker
├── 内存计数器（AtomicLong）:
│   ├── totalCompare   — 总对比次数
│   ├── bothFound      — 两通道都找到
│   ├── vectorOnly     — 仅向量找到
│   ├── ruleOnly       — 仅规则找到
│   └── neither        — 都没找到
├── 内存缓存（最近 50 条）:
│   └── List<Map> recent  — 线程安全（synchronized）
└── DB 持久化:
    └── eval_rag_compare_log 表
        ├── bizId, sceneCode, indexCode, indexName, dataValue
        ├── vectorResults（JSON）, ruleResults（JSON）
        ├── vectorSimilarities（JSON）
        ├── vectorHit（0/1）, ruleHit（0/1）
        └── groundTruthRel（JSON，仅跑批模式）
```

### 5.3 影子模式的运行时安全性

- **不阻塞主流程**：向量/规则检索异常各自独立 try-catch
- **DB 写入非致命**：写入失败只 warn 不抛异常
- **降级透明**：Qdrant 不可用时自动退化为纯规则检索，`buildFewShot` 行为与 A3.2 之前完全一致

## 6. 跑批结果

### 6.1 汇总表

> 跑批时间: 2026-07-22 | 数据: 15 条标注查询 | 向量存储: Qdrant (Cosine)

| 指标 | 通道 | K=1 | K=3 | K=5 |
|------|------|-----|-----|-----|
| **Hit Rate** | 向量 | 20.0% | 33.3% | **53.3%** |
| | 规则 | 26.7% | 26.7% | 26.7% |
| **NDCG** | 向量 | 100% | 95.0% | 82.8% |
| | 规则 | 100% | 98.9% | 98.7% |

### 6.2 关键发现

#### 发现 1: Top-1 不可靠（HR@1=20%）

向量检索的第一名只有 20% 命中率。80% 的情况下，Qdrant 返回的"最相似案例"并不是人工认为最相关的。这意味着：

- **few-shot 注入 Top-1 案例有风险**——可能注入不相关的案例误导 LLM
- **缓解策略**：当前注入 Top-3，增加了命中概率（HR@3=33.3%，HR@5=53.3%）

#### 发现 2: 规则引擎多样性差

规则检索的 HR 在 K>1 后完全不变（26.7%→26.7%→26.7%）。原因是：

- `SimilarCaseService` 只检索同 `indexCode` 的记录
- 按得分接近度排序，Top-{2,3,4,5} 与 Top-1 高度相似
- **本质上是"返回最接近的那条 + 它的邻居"**，不是"返回不同类型的相关案例"

这验证了规则检索的核心局限：它能找到"数值最接近的"，但找不到"语义不同但同样有参考价值的"。

#### 发现 3: 向量检索的召回率尚可（HR@5=53.3%）

超过一半的查询在 Top-5 中能命中。对于 168 条数据 + 512 维 bge-small-zh 来说，这个结果是合理的。改善方向：

- 换更大模型（bge-large-zh，1024 维）→ 预期 HR@5 提升到 65-70%
- 增加历史数据量（当前仅 168 条，很多查询根本没有相关案例在索引中）

#### 发现 4: 边界查询全部失败

gt-009 ~ gt-013（零值、极端值、无数据、稀有指标、负值）两通道都返回空或低相关。这验证了设计假设——没有标注相关案例的情况下，系统应该找不到。**"正确地找不到"和"正确地找到"同样重要。**

但这也暴露了一个工程问题：稀有指标（gt-012 `NEW_METRIC_RARE`）历史上不存在，向量检索只能靠指标名称的语义去匹配——对于完全陌生的指标名称，这不可靠。需要冷启动策略：当检索结果为空时，随机抽取同 diffLevel 的案例作为 fallback。

#### 发现 5: 双通道互补

| 场景 | 向量优势 | 规则优势 |
|------|---------|---------|
| 数值相似但语义不同 | ❌ 可能误导 | ✅ 精确匹配 |
| 语义相似但数值不同 | ✅ 能召回 | ❌ 找不到 |
| 跨指标联想 | ✅ 语义相近 | ❌ 跨不了 indexCode |
| 同指标精确对比 | ✅ 能召回 | ✅ 能找到 |

**结论**：向量做"广度召回"，规则做"精度锚定"。当前"向量优先、规则降级"的策略正确，但更好的做法是混合：向量召回 Top-5 + 规则过滤同 indexCode → 合并去重取 Top-3。

## 7. 测试

### 7.1 RagQualityServiceTest（8 个单元测试）

```
hrAtK_allHit              → HR@1=2/3, HR@3=3/3
hrAtK_allMiss             → HR@1=0, HR@3=0
hrAtK_emptyInput          → 空输入返回空 Map

ndcgAtK_perfectOrder      → rel=[1,1,1], NDCG@3=1.0
ndcgAtK_suboptimalOrder   → rel=[0,1,0], NDCG@3≈0.63
ndcgAtK_allIrrelevant     → rel=[0,0,0], NDCG@3=1.0 (DCG=IDCG=0 → 1.0)
ndcgAtK_multiQuery        → 2 条查询平均 ≈0.815
ndcgAtK_kLargerThanResults → K=5 > 结果数=2, 正常计算
```

所有测试通过，覆盖正常、边界、异常三类场景。

## 8. 已知限制与后续改进

### 8.1 当前限制

| 限制 | 影响 | 优先级 |
|------|------|--------|
| 标注数据仅 15 条 | 统计显著性不足，不能做细粒度分析 | P1 |
| 二元相关 | 无法区分"非常相关"和"有点相关" | P2 |
| 单一指标覆盖 | 只有 3 个指标（COST_DEV/FILL_RATE/ABNORM_CNT） | P2 |
| 标注者偏见 | 一个人标的，可能有认知盲区 | P3 |
| 规则引擎 hit rate 不变 | 暴露了多样性问题但跑批本身没解决 | P1 |

### 8.2 后续改进方向

1. **标注数据扩容**：目标 50 条（A1.3 计划中），跨 5+ 指标，引入多级相关度
2. **混合检索**：向量召回 Top-N → 规则过滤同 indexCode → RRF 合并排序
3. **换 embedding 模型**：bge-small-zh → bge-large-zh（1024 维），预期 HR@5 提 10-15pp
4. **冷启动策略**：向量检索为空时，按 diffLevel 随机抽取同指标案例
5. **可视化看板**：对接 Dashboard，展示实时对比数据（vector vs rule hit rate 趋势）
